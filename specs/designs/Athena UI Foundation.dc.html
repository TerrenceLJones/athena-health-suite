19.8Who created whom, and when
Every account creation and every reissue writes a FHIR AuditEvent recording the actor, the role assigned, the target account, the organization, and the timestamp.
Role authority enforced server-side
An Admin requesting role=admin or role=account_owner is rejected with a 403 — the UI omission is defense in depth, not the control.
Organization isolation
Viewing or modifying an account in another organization via direct API/URL is rejected with a 403.
Reissue invalidates the prior password
Reissuing generates a new one-time password and invalidates the previous one, even if it was never used.
First-login change required
The temporary password works once; the user must set a complexity-compliant password before reaching the dashboard, after which the temp password can't be reused.
19.9Guarantees behind the UI
The screens above are the visible half; these server-side rules are the enforced half. UI affordances never stand in for authorization.
